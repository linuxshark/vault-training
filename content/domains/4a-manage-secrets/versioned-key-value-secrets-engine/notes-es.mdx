**Docs:** https://www.vaultproject.io/docs/secrets

- Plugins utilizados por Vault para gestionar datos sensibles
- Almacenar, generar o encriptar datos
- Categorías amplias: motores de secretos **Dinámicos** y **Estáticos**
- Categorías generales de motores de secretos:
  - **Cloud** - AWS, Azure, GCP, etc.
  - **Database** - MSSQL, PostgreSQL, MySQL, etc.
  - **Internal** - Key/Value, Identity, Transit, etc.
  - **Identity** - Active Directory, LDAP, etc.
  - **Certificate** - SSH, PKI, etc.
  - **Tokens** - Consul, Nomad
- Todos los motores se habilitan en la ruta `/sys/mounts`
- Al habilitar, si no se proporciona una ruta, se establecerá por defecto el nombre del motor
  - **Ejemplo:** `vault secrets enable aws` se habilita y monta en `/sys/mounts/aws`
- Los motores de secretos pueden moverse
  - **ADVERTENCIA:** Si se mueven, se revocarán todos los arrendamientos existentes
  - Puede afectar las políticas (que se basan en rutas especificadas)
- Los motores pueden sintonizarse y configurarse
  - La configuración de sintonización es la misma para todos los motores de secretos (por ejemplo, descripción, TTL máximo de arrendamiento, etc.)
  - La configuración específica es propia de todo el motor de secretos

> **Nota:** Los motores de secretos específicos no son necesarios para el examen de certificación, solo cómo usarlos en general.

### Tipos de secretos

- **Secretos estáticos**

  - Almacenar datos existentes de forma segura
  - Ya tienes estos datos, y Vault debe gestionarlos y tener acceso a ellos
  - Gestión manual del ciclo de vida
  - Cargar manualmente nuevas versiones del secreto en un motor de secretos
  - **Ejemplo:**
    - El motor de secretos Key/Value (`kv`) puede almacenar algún valor estático bajo alguna clave

- **Secretos dinámicos**
  - Generar datos de secretos bajo demanda
  - _Arrendamiento_ emitido para cada secreto (TTL) que puede expirar
  - Gestión automática del ciclo de vida de secretos
  - La mayoría de los motores de secretos son de naturaleza dinámica

### Trabajar con motores de secretos

- Listar motores de secretos existentes
  - `vault secrets list`
  - `-detailed` para más información sobre cada motor
- Habilitar un nuevo motor de secretos
  - `vault secrets enable [OPTIONS] TYPE`
  - **Ejemplo:** Motor de secretos Key-value versión 2
    - `vault secrets enable -version=2 kv`
  - **Ejemplo:** Motor de secretos de base de datos
    - `vault secrets enable -path=CompanyData database`
- Sintonizar la configuración de un motor de secretos
  - `vault secrets tune [OPTIONS] PATH`
  - **Ejemplo:**
    - `vault secrets tune -max-lease-ttl=72h kv`
    - `vault secrets tune -description="Our cool database engine" CompanyData`
- Mover un motor de secretos
  - Revocará todos los arrendamientos incluidos
  - Podría afectar las restricciones de política
  - `vault secrets move [OPTIONS] SOURCE DESTINATION`
  - **Ejemplo:**
    - `vault secrets move CompanyData CompanyDataPrivate`
- Deshabilitar un motor de secretos
  - `vault secrets disable [OPTIONS] PATH`
  - **Ejemplo:**
    - `vault secrets disable CompanyData`
- Mostrar ayuda del motor de secretos
  - `vault path-help /sys/mounts/<engine>`
  - **Ejemplo:**
    - `vault path-help /sys/mounts/CompanyData`

### Envolvimiento de respuestas

**Docs:** https://www.vaultproject.io/docs/concepts/response-wrapping
**Tutorial:** https://learn.hashicorp.com/tutorials/vault/cubbyhole-response-wrapping

- Proporciona un mecanismo para compartir información entre múltiples entornos
- Vault toma la respuesta que habría enviado a un cliente e la inserta en el `cubbyhole` con un token de una sola vez, devolviendo ese token de una sola vez al cliente en su lugar
- La respuesta se envuelve mediante ese token `cubbyhole`
- Recuperar los datos reales dentro del `cubbyhole` requiere una operación de desenvuelto
  - **CLI:** `vault unwrap <wrapping token>`
  - **UI:** Tools > Unwrap > Enter token > Unwrap data
- El envolvimiento de respuesta proporciona:
  - El valor transmitido no es el secreto real sino una referencia al secreto real (token de envolvimiento de respuesta)
  - Solo una única parte puede desenvolver el token y ver qué hay dentro
  - Limita la vida útil de la exposición del secreto. Si el cliente no logra desenvolver el token, el token puede expirar rápidamente
- Creación de token envuelto
  - Usar la opción de bandera `-wrap-ttl=[DURATION]` vía CLI
  - Usar el encabezado `X-Vault-Wrap-TTL` vía API
- Cuando se solicita un envolvimiento de respuesta...
  - Se genera un token de **una sola vez**
  - La respuesta original se almacena en el cubbyhole del token de una sola vez
  - Se devuelve una nueva respuesta con un token de una sola vez

> **NOTA:** El cliente no necesita autenticarse en vault al intentar desenvolver, obtiene acceso instantáneo al secreto una sola vez

**Usando CLI**

- Envolver una respuesta de cualquier comando
  - `vault command -wrap-ttl=[DURATION] PATH`
  - El token resultante (`wrapping_token`) se pasa a la entidad que necesita recuperar esa respuesta envuelta
    **Ejemplo:** Crear un secreto y envolver la salida del valor para la clave especificada
    - `vault kv put kv/some_data some_key=some_value`
    - `vault kv get -wrap-ttl=1h kv/some_data`
    - Salida:
      - ```text
        Key                              Value
        ---                              -----
        wrapping_token:                  s.WWSuR0GwvjJgscyqS8rffAKL
        wrapping_accessor:               XwhdoFsKzG2EEoyPQaYTZtj9
        wrapping_token_ttl:              1h
        wrapping_token_creation_time:    2022-01-21 22:28:21.2741588 +0000 UTC
        wrapping_token_creation_path:    kv/data/some_data
        ```
- Desenvolver usando el token de envolvimiento emitido
    - `vault unwrap [OPTIONS] [TOKEN]`
    - **Ejemplo:** Desenvolver la respuesta envuelta anterior
       - `vault unwrap -field=data -format=json s.WWSuR0GwvjJgscyqS8rffAKL`
       - Salida:
           - ```json
             { "some_key": "some_value" }
              ```

**Usando API**

Similar a CLI, pero usando la API en el siguiente formato:

- Envolver una respuesta de cualquier comando. Resultará en un `wrapping_token`
  - ```bash
    curl --header "X-Vault-Wrap-TTL: <TTL>" \      # <-- NOTA
        --header "X-Vault-Token: <TOKEN>" \
        --request <HTTP_VERB> \
        --data '<PARAMETERS>' \
        <VAULT_ADDRESS>/v1/<ENDPOINT>
    ```
- Desenvolver usando el token emitido
  - ```bash
    curl --header "X-Vault-Token: <WRAPPING TOKEN>" \
        --request POST \
        <VAULT_ADDRESS>/v1/sys/wrapping/unwrap
    ```

### Motor de secretos: Cubbyhole

**Docs:** https://www.vaultproject.io/docs/secrets/cubbyhole

- Almacena secretos arbitrarios dentro de Vault, con espacio de nombres para un token
- Las rutas tienen alcance por token
- Ningún token puede acceder a la ruta cubbyhole de otro token
- Cada cubículo dentro del cubbyhole se crea por token de servicio
- El cubículo solo es accesible por ese token
- Habilitado por defecto
- No puede ser deshabilitado ni movido
- Sin control de versiones para secretos
- **El token raíz no tiene acceso al cubbyhole, solo su token puede acceder a él**
- Uso:
  - Escribir: `vault write cubbyhole/my-secret my-value=s3cr3t`
  - Leer: `vault read cubbyhole/my-secret`
  - API: Lectura vía curl
    - ```bash
      curl \
      	--header "X-Vault-Token: <TOKEN HERE>" \
      	http://127.0.0.1:8200/v1/cubbyhole/my-secret
      ```

### Motor de secretos: Identity

**Docs:** https://www.vaultproject.io/docs/secrets/identity
**Tutorial:** https://learn.hashicorp.com/tutorials/vault/identity

- Mantiene y controla clientes que son reconocidos por Vault
- Los "clientes" son usuarios o aplicaciones que han sido autenticados por Vault
- El motor de identidad rastrea esos clientes internamente dentro de Vault
- Habilitado/Montado por defecto
- No puede ser deshabilitado, movido o duplicado
- No se pueden habilitar múltiples instancias/rutas de los motores de identidad
- Cada cliente es una `Entity`
  - Entrada dentro del motor de identidad que representa un cliente
- Cualquier entidad puede tener múltiples `Aliases` (formas de autenticarse)
  - Por ejemplo, un usuario único que tiene cuentas tanto en GitHub como en LDAP puede ser mapeado a una única entidad en Vault que tiene 2 alias, uno de tipo GitHub y otro de tipo LDAP.
- Puede colocar entidades en diferentes grupos para administrar asignaciones de políticas a entidades
- Mantiene internamente a los clientes que son reconocidos por Vault
- Vincula identidades a tokens
- Contiene entidades, alias y grupos
- Puede asociar políticas de Vault a entidades o grupos
  - Las políticas de token típicas se asignan y evalúan en el momento de creación y renovación **(estáticas)**
  - Las políticas de identidad se asignan a entidad o grupo, se evalúan en cada solicitud **(dinámicas)**:
- Tres componentes principales
  1. **Entity**
      - Representa un cliente (usuario, máquina, etc)
      - Vinculada a un token
      - Ejemplo: Crear una entidad
        - `vault write identity/entity name=my_entity`
  2. **Alias**
      - Representa una forma de objetos de método de autenticación (es decir, aws, github, ldap, etc)
      - Vinculada a una entidad (o grupo)
  3. **Group**
      - Grupos organizados internamente
        - Grupo creado internamente por Vault
        - Puede contener otros grupos y entidades
      - Grupos organizados externamente
        - Métodos de autenticación como LDAP, Okta o GitHub
        - Usar alias para membresía
      - Puede tener entidades u otros grupos como miembros
      - Puede crear una jerarquía de grupos, con herencia de políticas

### Motor de secretos: Database

**Docs:** https://www.vaultproject.io/docs/secrets/databases

- Credenciales generadas dinámicamente con la propia base de datos
- Las credenciales tienen corta duración
- Las aplicaciones/servicios solicitan credenciales desde vault, que a su vez las genera a través de la base de datos
- También admite roles estáticos, un mapeo 1 a 1 entre roles de vault y nombre de usuario/contraseña

### Motor de secretos: Key Value (KV)

**Docs:** https://www.vaultproject.io/docs/secrets/kv

Hay dos versiones de este motor

- **Key Value Versión 1**

  - Sin control de versiones, el último gana
  - Más rápido con menos llamadas de almacenamiento
  - Los elementos eliminados se pierden para siempre
  - Puede actualizarse a la versión 2
  - Versión predeterminada en la creación

- **Key Value Versión 2**

  - Control de versiones de secretos anteriores
  - Posiblemente menos rendimiento
  - Elementos eliminados y metadatos retenidos, pueden recuperarse
  - No puede rebajarse a la versión 1
  - Puede especificarse en la creación

- Uso

  - **Docs:**: https://www.vaultproject.io/docs/secrets/kv/kv-v2#usage
  - Obtener ayuda: `vault kv --help`
  - Crear/Escribir un secreto:
    - `vault kv put [OPTIONS] KEY [DATA KEY=VALUE]`
    - **Ejemplo:** `vault kv put kv/apikeys/d101 token=1234567890`
    - **Ejemplo:** `vault kv put kv/apikeys/d101 @some_file.txt`
  - Leer un secreto:
    - `vault kv get [OPTIONS] KEY`
    - **Ejemplo:** `vault kv get -version=2 kv/apikeys/d101`
    - **Ejemplo:** `vault kv get -field=token kv/apikeys/d101`
    - **Ejemplo:** `vault kv get -format=json kv/apikeys/d101`
  - Listar secretos:
    - `vault kv list [OPTIONS] PATH`
    - **Ejemplo:** `vault kv list kv/apikeys`
  - Eliminar un secreto:
    - `vault kv delete [OPTIONS] KEY`
    - **Ejemplo:** `vault kv delete -versions=1 kv/apikeys/d101`
  - Destruir permanentemente un secreto:
    - `vault kv destroy [OPTIONS] KEY`
    - **Ejemplo:** `vault kv destroy kv/apikeys/d101`

- Ejemplo usando API REST:
  - Nota el `data` en la ruta
    - ```bash
      curl --header "X-Vault-Token: [VAULT TOKEN]" \
            http://127.0.0.1:8200/v1/kv/data/apitokens/d102
      ```
  - Obtener una versión específica de la clave-valor
    - ```bash
      curl --header "X-Vault-Token: [VAULT TOKEN]" \
            http://127.0.0.1:8200/v1/kv/data/apitokens/d102?version=1
      ```

### Motor de secretos: Transit

**Docs:** https://www.vaultproject.io/docs/secrets/transit
**Tutorial:** https://learn.hashicorp.com/tutorials/vault/eaas-transit

- Proporciona **encriptación como servicio**
- No almacena los datos enviados a este motor
- Solo se almacenan las claves encriptadas
- Acciones soportadas:
  - Encriptar y desencriptar datos
  - Puede firmar y verificar que los datos
  - Generar hashes
  - Crear bytes aleatorios
- Alivia la carga de encriptación/desencriptación adecuada
- Las claves de encriptación que utiliza para realizar todas las acciones se mantienen en el motor de secretos Transit
- La ruta predeterminada es `transit/`
