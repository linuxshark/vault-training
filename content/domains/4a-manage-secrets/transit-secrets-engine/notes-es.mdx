**Docs:** https://www.vaultproject.io/docs/secrets

- Plugins utilizados por Vault para gestionar datos sensibles
- Almacenar, generar o encriptar datos
- Amplias categorías: **Dynamic** y **Static** secret engines
- Categorías generales de Secrets engines:
  - **Cloud** - AWS, Azure, GCP, etc.
  - **Database** - MSSQL, PostgreSQL, MySQL, etc.
  - **Internal** - Key/Value, Identity, Transit, etc.
  - **Identity** - Active Directory, LDAP, etc.
  - **Certificate** - SSH, PKI, etc.
  - **Tokens** - Consul, Nomad
- Todos los engines están habilitados en la ruta `/sys/mounts`
- Al habilitar, si no se proporciona una ruta, será por defecto el nombre del engine
  - **Ejemplo:** `vault secrets enable aws` se habilita y monta en `/sys/mounts/aws`
- Los Secrets engines se pueden mover
  - **ADVERTENCIA:** Si se mueven, se revocarán todos los leases existentes
  - Puede afectar las políticas (que se basan en rutas especificadas)
- Los engines se pueden ajustar y configurar
  - Los ajustes de tuning son los mismos para todos los secrets engines (es decir, descripción, máximo lease TTL, etc.)
  - Los ajustes de configuración son específicos para todo el secrets engine

> **Nota:** Los secrets engines específicos no son necesarios para el examen de certificación, solo cómo usarlos de manera general.

### Tipos de Secrets

- **Static secrets**

  - Almacenar datos existentes de forma segura
  - Ya tienes estos datos, y Vault necesita gestionarlos y tener acceso a ellos
  - Gestión manual del ciclo de vida
  - Cargar manualmente nuevas versiones del secret en un secrets engine
  - **Ejemplo:**
    - El secrets engine Key/Value (`kv`) puede almacenar algún valor estático bajo alguna clave

- **Dynamic Secrets**
  - Generar datos de secretos bajo demanda
  - _Lease_ emitido para cada secreto (TTL) que puede expirar
  - Gestión automática del ciclo de vida de secretos
  - La mayoría de secrets engines son dinámicos por naturaleza

### Trabajar con Secrets Engines

- Listar secrets engines existentes
  - `vault secrets list`
  - `-detailed` para más información de cada engine
- Habilitar un nuevo secrets engine
  - `vault secrets enable [OPTIONS] TYPE`
  - **Ejemplo:** Secret engine Key-value versión 2
    - `vault secrets enable -version=2 kv`
  - **Ejemplo:** Secret engine Database
    - `vault secrets enable -path=CompanyData database`
- Ajustar configuración de un secrets engine
  - `vault secrets tune [OPTIONS] PATH`
  - **Ejemplo:**
    - `vault secrets tune -max-lease-ttl=72h kv`
    - `vault secrets tune -description="Our cool database engine" CompanyData`
- Mover un secrets engine
  - Revocará todos los leases incluidos
  - Podría afectar restricciones de políticas
  - `vault secrets move [OPTIONS] SOURCE DESTINATION`
  - **Ejemplo:**
    - `vault secrets move CompanyData CompanyDataPrivate`
- Deshabilitar un secrets engine
  - `vault secrets disable [OPTIONS] PATH`
  - **Ejemplo:**
    - `vault secrets disable CompanyData`
- Mostrar ayuda de secrets engine
  - `vault path-help /sys/mounts/<engine>`
  - **Ejemplo:**
    - `vault path-help /sys/mounts/CompanyData`

### Response Wrapping

**Docs:** https://www.vaultproject.io/docs/concepts/response-wrapping
**Tutorial:** https://learn.hashicorp.com/tutorials/vault/cubbyhole-response-wrapping

- Proporciona un mecanismo para compartir información entre muchos entornos
- Vault toma la respuesta que habría enviado a un cliente e la inserta en `cubbyhole` con un token de uso único, devolviendo ese token de uso único al cliente en su lugar
- La respuesta es envuelta por ese token `cubbyhole`
- Recuperar los datos reales dentro de `cubbyhole` requiere una operación de unwrap
  - **CLI:** `vault unwrap <wrapping token>`
  - **UI:** Tools > Unwrap > Enter token > Unwrap data
- Response wrapping proporciona:
  - El valor transmitido no es el secreto real sino una referencia al secreto real (response-wrapping token)
  - Solo una única parte puede desenrollar el token y ver qué hay adentro
  - Limita el tiempo de vida de exposición del secreto. Si el cliente falla al desenrollar el token, este puede expirar rápidamente
- Creación de token envuelto
  - Usa la opción `-wrap-ttl=[DURATION]` a través de CLI
  - Usa el header `X-Vault-Wrap-TTL` a través de API
- Cuando se solicita un response wrap ...
  - Se genera un token de **uso único**
  - La respuesta original se almacena en el cubbyhole del token de uso único
  - Se devuelve una nueva respuesta con un token de uso único

> **NOTA:** El cliente no tiene que autenticarse en vault al intentar desenrollar, obtiene acceso instantáneo al secreto una única vez

**Usando CLI**

- Envolver una respuesta de cualquier comando
  - `vault command -wrap-ttl=[DURATION] PATH`
  - El token resultante (`wrapping_token`) se pasa a la entidad que necesita recuperar esa respuesta envuelta
    **Ejemplo:** Crear un secreto y envolver la salida del valor para la clave especificada
    - `vault kv put kv/some_data some_key=some_value`
    - `vault kv get -wrap-ttl=1h kv/some_data`
    - Salida:
      - ```text
        Key                              Value
        ---                              -----
        wrapping_token:                  s.WWSuR0GwvjJgscyqS8rffAKL
        wrapping_accessor:               XwhdoFsKzG2EEoyPQaYTZtj9
        wrapping_token_ttl:              1h
        wrapping_token_creation_time:    2022-01-21 22:28:21.2741588 +0000 UTC
        wrapping_token_creation_path:    kv/data/some_data
        ```
- Desenrollar usando el token de wrap emitido
    - `vault unwrap [OPTIONS] [TOKEN]`
    - **Ejemplo:** Desenrollar la respuesta previamente envuelta
       - `vault unwrap -field=data -format=json s.WWSuR0GwvjJgscyqS8rffAKL`
       - Salida:
           - ```json
             { "some_key": "some_value" }
              ```

**Usando API**

Similar a CLI, pero usando la API en el siguiente formato:

- Envolver una respuesta de cualquier comando. Resultará en un `wrapping_token`
  - ```bash
    curl --header "X-Vault-Wrap-TTL: <TTL>" \      # <-- NOTA
        --header "X-Vault-Token: <TOKEN>" \
        --request <HTTP_VERB> \
        --data '<PARAMETERS>' \
        <VAULT_ADDRESS>/v1/<ENDPOINT>
    ```
- Desenrollar usando el token emitido
  - ```bash
    curl --header "X-Vault-Token: <WRAPPING TOKEN>" \
        --request POST \
        <VAULT_ADDRESS>/v1/sys/wrapping/unwrap
    ```

### Secrets Engine: Cubbyhole

**Docs:** https://www.vaultproject.io/docs/secrets/cubbyhole

- Almacena secretos arbitrarios dentro de Vault, namespaceados a un token
- Las rutas están limitadas por token
- Ningún token puede acceder a la ruta cubbyhole de otro token
- Cada cubbyholes se crea por token de servicio
- El cubbyhole es solo accesible por ese token
- Habilitado por defecto
- No se puede deshabilitar ni mover
- Sin versionado para secretos
- **El token root no tiene acceso al cubbyhole, solo su token puede acceder a él**
- Uso:
  - Escribir: `vault write cubbyhole/my-secret my-value=s3cr3t`
  - Leer: `vault read cubbyhole/my-secret`
  - API: Leer a través de curl
    - ```bash
      curl \
      	--header "X-Vault-Token: <TOKEN HERE>" \
      	http://127.0.0.1:8200/v1/cubbyhole/my-secret
      ```

### Secrets Engine: Identity

**Docs:** https://www.vaultproject.io/docs/secrets/identity
**Tutorial:** https://learn.hashicorp.com/tutorials/vault/identity

- Mantiene y hace seguimiento de los clientes reconocidos por Vault
- Los "clientes" son usuarios o aplicaciones que han sido autenticados por Vault
- El secrets engine Identity rastrea esos clientes internamente dentro de Vault
- Habilitado/Montado por defecto
- No se puede deshabilitar, mover o duplicar
- No se pueden habilitar múltiples instancias/rutas del secrets engine Identity
- Cada cliente es una `Entity`
  - Entrada dentro del secrets engine Identity que representa a un cliente
- Cualquier entity puede tener múltiples `Aliases` (formas de autenticarse)
  - Por ejemplo, un usuario único que tiene cuentas tanto en GitHub como en LDAP, puede ser mapeado a una única entity en Vault que tiene 2 aliases, uno de tipo GitHub y otro de tipo LDAP.
- Se pueden colocar entities en diferentes grupos para gestionar asignaciones de políticas a entities
- Mantiene internamente los clientes reconocidos por Vault
- Vincula identidades a tokens
- Contiene entities, aliases y grupos
- Se pueden asociar Vault policies a entities o grupos
  - Las políticas típicas de token se asignan y se evalúan en el momento de creación y renovación **(estáticas)**
  - Las políticas de Identity se asignan a una entity o grupo, se evalúan en cada request **(dinámicas)**:
- Tres componentes principales
  1. **Entity**
      - Representa a un cliente (usuario, máquina, etc)
      - Vinculada a un token
      - Ejemplo: Crear una entity
        - `vault write identity/entity name=my_entity`
  2. **Alias**
      - Representa una forma de objetos de método de autenticación (es decir, aws, github, ldap, etc)
      - Vinculada a una entity (o grupo)
  3. **Group**
      - Grupos organizados internamente
        - Grupo creado internamente por Vault
        - Puede contener otros grupos y entities
      - Grupos organizados externamente
        - Métodos de autenticación como LDAP, Okta o GitHub
        - Usar aliases para membresía
      - Puede tener entities u otros grupos como miembros
      - Puede crear una jerarquía de grupos, con herencia de políticas

### Secrets Engine: Database

**Docs:** https://www.vaultproject.io/docs/secrets/databases

- Credenciales generadas dinámicamente con la base de datos misma
- Las credenciales tienen corta duración
- Las aplicaciones/Servicios solicitan credenciales a vault, que a su vez las genera a través de la base de datos
- También soporta static roles, un mapeo 1-a-1 entre vault roles y username/password

### Secrets Engine: Key Value (KV)

**Docs:** https://www.vaultproject.io/docs/secrets/kv

Hay dos versiones de este engine

- **Key Value Version 1**

  - Sin versionado, el último valor gana
  - Más rápido con menos llamadas de almacenamiento
  - Los elementos eliminados se pierden para siempre
  - Se puede actualizar a versión 2
  - Versión por defecto al crear

- **Key Value Version 2**

  - Versionado de secretos previos
  - Posiblemente menos rendimiento
  - Elementos eliminados y metadatos retenidos, se pueden recuperar
  - No se puede degradar a versión 1
  - Se puede especificar al crear

- Uso

  - **Docs:**: https://www.vaultproject.io/docs/secrets/kv/kv-v2#usage
  - Obtener ayuda: `vault kv --help`
  - Crear/Escribir un secreto:
    - `vault kv put [OPTIONS] KEY [DATA KEY=VALUE]`
    - **Ejemplo:** `vault kv put kv/apikeys/d101 token=1234567890`
    - **Ejemplo:** `vault kv put kv/apikeys/d101 @some_file.txt`
  - Leer un secreto:
    - `vault kv get [OPTIONS] KEY`
    - **Ejemplo:** `vault kv get -version=2 kv/apikeys/d101`
    - **Ejemplo:** `vault kv get -field=token kv/apikeys/d101`
    - **Ejemplo:** `vault kv get -format=json kv/apikeys/d101`
  - Listar secretos:
    - `vault kv list [OPTIONS] PATH`
    - **Ejemplo:** `vault kv list kv/apikeys`
  - Eliminar un secreto:
    - `vault kv delete [OPTIONS] KEY`
    - **Ejemplo:** `vault kv delete -versions=1 kv/apikeys/d101`
  - Destruir permanentemente un secreto:
    - `vault kv destroy [OPTIONS] KEY`
    - **Ejemplo:** `vault kv destroy kv/apikeys/d101`

- Ejemplo usando REST API:
  - Nota el `data` en la ruta
    - ```bash
      curl --header "X-Vault-Token: [VAULT TOKEN]" \
            http://127.0.0.1:8200/v1/kv/data/apitokens/d102
      ```
  - Obtener una versión específica del key value
    - ```bash
      curl --header "X-Vault-Token: [VAULT TOKEN]" \
            http://127.0.0.1:8200/v1/kv/data/apitokens/d102?version=1
      ```

### Secrets Engine: Transit

**Docs:** https://www.vaultproject.io/docs/secrets/transit
**Tutorial:** https://learn.hashicorp.com/tutorials/vault/eaas-transit

- Proporciona **encryption as a service**
- No almacena los datos enviados a este engine
- Solo se almacenan las claves encriptadas
- Acciones soportadas:
  - Encriptar y desencriptar datos
  - Puede firmar y verificar que los datos
  - Generar hashes
  - Crear bytes aleatorios
- Alivia la carga de encriptación/desencriptación adecuada
- Las claves de encriptación que utiliza para realizar todas las acciones se mantienen en el secrets engine Transit
- La ruta por defecto es `transit/`
- Operaciones
  - Habilitar engine
  - Añadir claves criptográficas
  - Ver claves y versiones
  - Encriptar datos con clave
  - Desencriptar datos con clave
  - Rotar y gestionar versiones de claves
- Características de claves criptográficas
  - Tipos básicos: AES, ChaCha20, ED25519, ECDSA, RSA, etc.
  - El tipo de clave determina las acciones soportadas
  - Encriptación convergente
    - Determinista - Independientemente de la versión de clave, el valor
