**Docs:** https://www.vaultproject.io/docs/secrets

- Plugins utilizados por Vault para manejar datos sensibles
- Almacenar, generar o encriptar datos
- Amplias categorías: **Dynamic** y **Static** secret engines
- Categorías generales de Secrets engine:
  - **Cloud** - AWS, Azure, GCP, etc.
  - **Database** - MSSQL, PostgreSQL, MySQL, etc.
  - **Internal** - Key/Value, Identity, Transit, etc.
  - **Identity** - Active Directory, LDAP, etc.
  - **Certificate** - SSH, PKI, etc.
  - **Tokens** - Consul, Nomad
- Todos los engines se habilitan en la ruta `/sys/mounts`
- Al habilitar, si no se proporciona una ruta, por defecto será el nombre del engine
  - **Ejemplo:** `vault secrets enable aws` se habilita y monta en `/sys/mounts/aws`
- Los Secrets engines se pueden mover
  - **ADVERTENCIA:** Si se mueven, se revocarán todos los leases existentes
  - Puede afectar las políticas (que se basan en rutas especificadas)
- Los engines se pueden ajustar y configurar
  - La configuración de ajuste es igual para todos los secrets engines (es decir, descripción, max lease TTL, etc.)
  - La configuración específica es única para cada secrets engine

> **Nota:** Los Secrets engines específicos no son necesarios para el examen de certificación, solo
> cómo usarlos en general.

### Tipos de Secretos

- **Secretos estáticos**

  - Almacenar datos existentes de forma segura
  - Ya tienes estos datos, y Vault necesita gestionarlos y acceder a ellos
  - Gestión manual del ciclo de vida
  - Cargar manualmente nuevas versiones del secreto en un secrets engine
  - **Ejemplo:**
    - El secrets engine Key/Value (`kv`) puede almacenar algún valor estático bajo alguna clave

- **Secretos Dinámicos**
  - Generar datos de secretos bajo demanda
  - Se emite un _lease_ para cada secreto (TTL) que puede expirar
  - Gestión automática del ciclo de vida de secretos
  - La mayoría de los secrets engines son de naturaleza dinámica

### Trabajar con Secrets Engines

- Listar secrets engines existentes
  - `vault secrets list`
  - `-detailed` para más información sobre cada engine
- Habilitar un nuevo secrets engine
  - `vault secrets enable [OPTIONS] TYPE`
  - **Ejemplo:** Secrets engine Key-value versión 2
    - `vault secrets enable -version=2 kv`
  - **Ejemplo:** Secrets engine de base de datos
    - `vault secrets enable -path=CompanyData database`
- Ajustar la configuración de un secrets engine
  - `vault secrets tune [OPTIONS] PATH`
  - **Ejemplo:**
    - `vault secrets tune -max-lease-ttl=72h kv`
    - `vault secrets tune -description="Our cool database engine" CompanyData`
- Mover un secrets engine
  - Revocará todos los leases incluidos
  - Podría afectar las restricciones de políticas
  - `vault secrets move [OPTIONS] SOURCE DESTINATION`
  - **Ejemplo:**
    - `vault secrets move CompanyData CompanyDataPrivate`
- Deshabilitar un secrets engine
  - `vault secrets disable [OPTIONS] PATH`
  - **Ejemplo:**
    - `vault secrets disable CompanyData`
- Mostrar ayuda del secrets engine
  - `vault path-help /sys/mounts/<engine>`
  - **Ejemplo:**
    - `vault path-help /sys/mounts/CompanyData`

### Response Wrapping

**Docs:** https://www.vaultproject.io/docs/concepts/response-wrapping
**Tutorial:** https://learn.hashicorp.com/tutorials/vault/cubbyhole-response-wrapping

- Proporciona un mecanismo para compartir información entre muchos entornos
- Vault toma la respuesta que hubiera enviado a un cliente e la inserta en el `cubbyhole` con
  un token de un solo uso, devolviendo ese token de un solo uso al cliente en su lugar
- La respuesta se envuelve mediante ese token del `cubbyhole`
- Recuperar los datos reales dentro del `cubbyhole` requiere una operación unwrap
  - **CLI:** `vault unwrap <wrapping token>`
  - **UI:** Tools > Unwrap > Enter token > Unwrap data
- Response wrapping proporciona:
  - El valor transmitido no es el secreto real, sino una referencia al secreto real (response-wrapping token)
  - Solo una parte puede desenvovler el token y ver qué hay dentro
  - Limita la vida útil de la exposición del secreto. Si el cliente falla al desenvovler el token, el token puede expirar rápidamente
- Creación de token envuelto
  - Usar la bandera de opción `-wrap-ttl=[DURATION]` vía CLI
  - Usar el header `X-Vault-Wrap-TTL` vía API
- Cuando se solicita un response wrap ...
  - Se genera un token de **un solo uso**
  - La respuesta original se almacena en el cubbyhole del token de un solo uso
  - Se devuelve una nueva respuesta con un token de un solo uso

> **NOTA:** El cliente no tiene que autenticarse en Vault al intentar desenvovler, obtiene acceso instantáneo al secreto una sola vez

**Usando CLI**

- Envolver la respuesta de cualquier comando
  - `vault command -wrap-ttl=[DURATION] PATH`
  - El token resultante (`wrapping_token`) se pasa a la entidad que necesita recuperar esa respuesta envuelta
    **Ejemplo:** Crear un secreto y envolver la salida del valor de la clave especificada
    - `vault kv put kv/some_data some_key=some_value`
    - `vault kv get -wrap-ttl=1h kv/some_data`
    - Salida:
      - ```text
        Key                              Value
        ---                              -----
        wrapping_token:                  s.WWSuR0GwvjJgscyqS8rffAKL
        wrapping_accessor:               XwhdoFsKzG2EEoyPQaYTZtj9
        wrapping_token_ttl:              1h
        wrapping_token_creation_time:    2022-01-21 22:28:21.2741588 +0000 UTC
        wrapping_token_creation_path:    kv/data/some_data
        ```
- Desenvovler usando el token envuelto emitido
    - `vault unwrap [OPTIONS] [TOKEN]`
    - **Ejemplo:** Desenvovler la respuesta previamente envuelta
       - `vault unwrap -field=data -format=json s.WWSuR0GwvjJgscyqS8rffAKL`
       - Salida:
           - ```json
             { "some_key": "some_value" }
              ```

**Usando API**

Similar a CLI, pero usando la API en el siguiente formato:

- Envolver la respuesta de cualquier comando. Resultará en un `wrapping_token`
  - ```bash
    curl --header "X-Vault-Wrap-TTL: <TTL>" \      # <-- NOTA
        --header "X-Vault-Token: <TOKEN>" \
        --request <HTTP_VERB> \
        --data '<PARAMETERS>' \
        <VAULT_ADDRESS>/v1/<ENDPOINT>
    ```
- Desenvovler usando el token emitido
  - ```bash
    curl --header "X-Vault-Token: <WRAPPING TOKEN>" \
        --request POST \
        <VAULT_ADDRESS>/v1/sys/wrapping/unwrap
    ```

### Secrets Engine: Cubbyhole

**Docs:** https://www.vaultproject.io/docs/secrets/cubbyhole

- Almacena secretos arbitrarios dentro de Vault, dentro del espacio de nombres de un token
- Las rutas se limitan por token
- Ningún token puede acceder a la ruta del cubbyhole de otro token
- Cada cubby dentro del cubby hole se crea por token de servicio
- El cubby solo es accesible por ese token
- Se habilita por defecto
- No se puede deshabilitar ni mover
- Sin versiones para secretos
- **El token raíz no tiene acceso al cubbyhole, solo su token puede acceder a él**
- Uso:
  - Escribir: `vault write cubbyhole/my-secret my-value=s3cr3t`
  - Leer: `vault read cubbyhole/my-secret`
  - API: Lectura vía curl
    - ```bash
      curl \
      	--header "X-Vault-Token: <TOKEN HERE>" \
      	http://127.0.0.1:8200/v1/cubbyhole/my-secret
      ```

### Secrets Engine: Identity

**Docs:** https://www.vaultproject.io/docs/secrets/identity
**Tutorial:** https://learn.hashicorp.com/tutorials/vault/identity

- Mantiene un registro de los clientes que son reconocidos por Vault
- Los "clientes" son usuarios o aplicaciones que han sido autenticados por Vault
- El motor Identity rastrea a esos clientes internamente dentro de Vault
- Se habilita/monta por defecto
- No se puede deshabilitar, mover o duplicar
- No se pueden habilitar múltiples instancias/rutas de los motores de identidad
- Cada cliente es una `Entity`
  - Entrada dentro del motor de identidad que representa un cliente
- Cualquier entidad puede tener múltiples `Aliases` (formas de autenticarse)
  - Por ejemplo, un usuario único que tiene cuentas tanto en GitHub como en LDAP,
    puede asignarse a una sola entidad en Vault que tiene 2 aliases, uno
    de tipo GitHub y otro de tipo LDAP.
- Puede colocar entidades en diferentes grupos para gestionar asignaciones de políticas a entidades
- Mantiene internamente los clientes que son reconocidos por Vault
- Vincula identidades a tokens
- Contiene entidades, aliases y grupos
- Puede asociar políticas de Vault a entidades o grupos
  - Las políticas de token típicas se asignan y evalúan en el tiempo de creación y renovación **(estáticas)**
  - Las políticas de identidad se asignan a una entidad o grupo, se evalúan en cada solicitud **(dinámicas)**:
- Tres componentes principales
  1. **Entity**
      - Representa un cliente (usuario, máquina, etc)
      - Vinculada a un token
      - Ejemplo: Crear una entidad
        - `vault write identity/entity name=my_entity`
  2. **Alias**
      - Representa una forma de objetos del método de autenticación (es decir, aws, github, ldap, etc)
      - Vinculada a una entidad (o grupo)
  3. **Group**
      - Grupos organizados internamente
        - Grupo creado internamente por Vault
        - Puede contener otros grupos y entidades
      - Grupos organizados externamente
        - Métodos de autenticación como LDAP, Okta o GitHub
        - Usar aliases para membresía
      - Puede tener entidades u otros grupos como miembros
      - Puede crear una jerarquía de grupos, con herencia de políticas

### Secret Engine: Database

**Docs:** https://www.vaultproject.io/docs/secrets/databases

- Las credenciales se generan dinámicamente con la base de datos misma
- Las credenciales tienen vida corta
- Las aplicaciones/servicios solicitan credenciales de Vault, que a su vez las genera a través de la base de datos
- También soporta roles estáticos, una asignación 1-a-1 entre roles de Vault y nombre de usuario/contraseña

### Secret Engine: Key Value (KV)

**Docs:** https://www.vaultproject.io/docs/secrets/kv

Hay dos versiones de este engine

- **Key Value Versión 1**

  - Sin versionado, la última clave gana
  - Más rápido con menos llamadas de almacenamiento
  - Los elementos eliminados desaparecen para siempre
  - Puede actualizarse a la versión 2
  - Versión por defecto en la creación

- **Key Value Versión 2**

  - Versionado de secretos anteriores
  - Posiblemente menos rendimiento
  - Elementos eliminados y metadatos retenidos, pueden recuperarse
  - No se puede degradar a la versión 1
  - Se puede especificar en la creación

- Uso

  - **Docs:**: https://www.vaultproject.io/docs/secrets/kv/kv-v2#usage
  - Obtener ayuda: `vault kv --help`
  - Crear/escribir un secreto:
    - `vault kv put [OPTIONS] KEY [DATA KEY=VALUE]`
    - **Ejemplo:** `vault kv put kv/apikeys/d101 token=1234567890`
    - **Ejemplo:** `vault kv put kv/apikeys/d101 @some_file.txt`
  - Leer un secreto:
    - `vault kv get [OPTIONS] KEY`
    - **Ejemplo:** `vault kv get -version=2 kv/apikeys/d101`
    - **Ejemplo:** `vault kv get -field=token kv/apikeys/d101`
    - **Ejemplo:** `vault kv get -format=json kv/apikeys/d101`
  - Listar secretos:
    - `vault kv list [OPTIONS] PATH`
    - **Ejemplo:** `vault kv list kv/apikeys`
  - Eliminar un secreto:
    - `vault kv delete [OPTIONS] KEY`
    - **Ejemplo:** `vault kv delete -versions=1 kv/apikeys/d101`
  - Destruir permanentemente un secreto:
    - `vault kv destroy [OPTIONS] KEY`
    - **Ejemplo:** `vault kv destroy kv/apikeys/d101`

- Ejemplo usando REST API:
  - Nota el `data` en la ruta
    - ```bash
      curl --header "X-Vault-Token: [VAULT TOKEN]" \
            http://127.0.0.1:8200/v1/kv/data/apitokens/d102
      ```
  - Obtener una versión específica del key value
    - ```bash
      curl --header "X-Vault-Token: [VAULT TOKEN]" \
            http://127.0.0.1:8200/v1/kv/data/apitokens/d102?version=1
      ```

### Secret Engine: Transit

**Docs:** https://www.vaultproject.io/docs/secrets/transit
**Tutorial:** https://learn.hashicorp.com/tutorials/vault/eaas-transit

- Proporciona **encriptación como servicio**
- No almacena los datos enviados a este engine
- Solo se almacenan las claves encriptadas
- Acciones soportadas:
  - Encriptar y desencriptar datos
  - Puede firmar y verificar datos
  - Generar hashes
  - Crear bytes aleatorios
- Alivia la carga de encriptación/desencriptación adecuada
- Las claves de encriptación que utiliza para realizar todas las acciones se mantienen en el secrets engine Transit
- La ruta por defecto es `transit/`
- Operaciones
  - Habilitar engine
  - Agregar claves criptográficas
  - Ver claves y versiones
  - Encriptar datos con clave
  - Desencriptar datos con clave
  - Rotar y gestionar versiones de claves
- Características de claves criptográficas
  -
